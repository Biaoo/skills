有时我和 AI 讨论出了一个看起来很有道理的原则。

它可能是某种设计理念、协作方式、表达风格、判断标准、系统边界，或者从另一个场景里抽象出来的经验。

但原则出现之后，我并不会马上觉得事情结束了。

我会有一种未完成感，或者说一种约束力追问：

```text
这个原则听起来对，但它会改变什么？
它能不能检查当前问题？
它能不能指出哪些地方不一致？
它能不能成为后续可复用的判断依据？
它能不能指导下一轮迭代？
它能不能告诉我什么算变好，什么算变坏？
```

这种追问说明，我关心的不是原则本身是否漂亮，而是它有没有开始产生约束力。

一个原则如果只是停留在语言里，它可能是一句正确但无效的话。

我暂时把这个过程叫做：原则降维。

## 当前理解

原则降维指的是：把一个抽象原则转化成当前问题中可观察、可检查、可复用、可优化的约束。

它不是简单地把原则变成一个任务。

更准确地说，它是在追问：

```text
这个原则如果是真的，
它会要求当前问题发生什么变化？
它能识别哪些偏差？
它应该以什么形式被保留？
它在当前问题里的优化目标是什么？
什么变化应该被奖励？
什么偏差应该被减少？
```

所以，原则降维不是让原则变低级，而是让原则重新接触现实。

它让一个高维判断从“听起来对”，变成可以影响当前问题的东西。

## 它从哪里冒出来

原则降维通常来自一种原则出现后的未完成感，或者说，对原则是否真的产生约束力的追问。

这种原则可能有两个来源。

第一种，是当前讨论中生成出来的原则。

比如我和 AI 讨论某个设计方向，逐渐形成一个原则：

```text
当前方案应该减少隐性复杂度，而不是只追求表面一致。
```

这个原则出现之后，我不会只停在“这个说法不错”。我会继续问：

```text
当前方案里哪些复杂度是隐性的？
哪些地方只是看起来统一，但实际上增加了维护负担？
这个原则以后还能不能继续作为判断依据？
什么样的改动算是在减少复杂度？
什么样的改动只是制造了新的抽象？
```

第二种，是从别的场景迁移过来的原则。

我可能在处理另一个事情时理解了某个背后原理，然后突然想到：

```text
这个原理是不是也适用于当前问题？
```

这时路径大概是：

```text
场景 A 的经验
-> 抽出背后原则
-> 想到当前问题 B
-> 判断 B 是否满足相似条件
-> 如果成立，把原则放进 B 里使用
```

这也是原则降维。因为真正的问题不是“这个类比有趣”，而是：

```text
这个从别处来的原则，能不能成为当前问题的判断标准？
它能不能改变当前方案里的某些选择？
它能不能指导当前问题的优化方向？
```

无论原则来自当前讨论，还是来自另一个场景，核心问题都是：

```text
它能不能进入当前现实，并产生约束力？
```

## 它如何工作

原则降维不是一步完成的。

它通常会经历几个动作：

```text
形成 / 迁移原则
-> 判断适用条件
-> 找影响范围
-> 扫描偏差
-> 判断是否沉淀
-> 转成优化目标
-> 形成奖励 / 损失信号
-> 做一次最小迭代
```

比如：

```text
原则：
当前方案应该减少隐性复杂度。

判断适用条件：
这个原则适用于当前问题吗？
当前问题的主要风险真的是复杂度吗？
有没有可能当前更重要的是交付速度、兼容性或表达清晰度？

找影响范围：
这个原则会影响哪些部分？
它影响结构、接口、状态管理、交互流程，还是长期维护方式？

扫描偏差：
哪些地方引入了看不见的维护成本？
哪些地方只是表面简洁，但实际增加了理解负担？
哪些地方为了统一而牺牲了局部清晰度？

判断是否沉淀：
这个原则是否只是当前问题中的临时判断？
还是可以成为后续类似问题的判断依据？
它需要哪些边界、例子和反例？

转成优化目标：
当前迭代优先减少隐性复杂度，而不是追求更多抽象。

形成奖励 / 损失信号：
奖励更清晰的边界、更少的隐藏依赖、更直接的状态流动。
减少过度抽象、隐式耦合、看似统一但难以维护的结构。

做一次最小迭代：
先选一个最明显的复杂点，尝试改成更清晰、更可解释的结构。
```

这里的关键不是把原则变成任务，而是让原则获得几个接口：

```text
检查接口：我能不能看出哪里不符合它？
沉淀接口：它要不要成为后续判断依据？
优化接口：它当前要优化什么？
反馈接口：我怎么知道改完以后更好了？
```

## 常见动作

下面这些不是固定流程，而是原则降维时经常出现的动作。

### 适用条件判断

如果原则来自当前讨论，它未必适用于所有场景。

如果原则来自另一个场景，它更需要先判断迁移条件。

```text
这个原则依赖什么前提？
当前问题是否满足这些前提？
两个场景是结构相似，还是只是表面相似？
这个原则在哪些情况下会失效？
哪些地方不能外推？
```

这一步防止我把一个有启发的原则过度推广。

### 影响范围追问

原则如果成立，应该能影响某些东西。

```text
基于这个原则，当前问题中哪些地方需要调整？
它影响的是结构、流程、表达、判断标准，还是后续迭代方式？
哪些地方只是局部要改？
哪些地方会影响整体方向？
这个原则会不会改变后续类似问题的处理方式？
```

这一步把原则从语言里拉出来，让它开始触碰现实。

### 偏差扫描

原则可以反过来扫描当前问题。

```text
哪些地方不符合这个原则？
哪些地方只是表面符合？
哪些地方和原则方向相反？
哪些地方虽然不完全符合，但因为成本或上下文可以暂时保留？
```

偏差扫描不是为了挑错，而是为了看清当前现实和原则之间的差异。

### 原则沉淀

如果一个原则反复出现，或者明显会影响后续判断，就需要考虑是否沉淀。

```text
这个原则只是当前场景有效，还是以后也能复用？
它应该以什么形式被保留？
它以后应该如何被重新调用？
它是判断标准、设计原则、操作规则，还是上下文约束？
它需要哪些边界、例子和反例？
```

沉淀让一次讨论变成后续可回看的认知资产。

### 优化目标化

当原则进入当前具体问题时，我需要问：现在到底要优化什么？

```text
基于这个原则和当前场景，当前问题的第一优化目标是什么？
什么变化算更好？
什么变化算变坏？
为了达成这个目标，我愿意牺牲什么？
后续迭代应该朝哪个方向收敛？
```

这是原则降维里很关键的一步。

因为没有优化目标，原则容易停留在价值判断里；有了优化目标，原则才开始指导实际迭代。

### 奖励 / 损失信号

有时我会用“奖励函数 / 损失函数”来描述当前迭代的抓手。

这里不一定是严格数学意义上的函数，而是一组判断信号：

```text
什么行为应该被奖励？
什么行为应该被减少？
什么变化说明方案变好了？
什么偏差说明方案变坏了？
什么信号说明我们正在接近想要的状态？
```

比如：

```text
原则：
当前方案应该减少隐性复杂度。

奖励信号：
边界更清晰
依赖更显性
状态流动更直接
局部行为更容易解释
后续修改成本更低

损失信号：
过度抽象
隐式耦合
表面统一但实际难懂
为了复用牺牲清晰度
短期省事但长期维护成本增加
```

奖励 / 损失信号让原则变成可以指导迭代的评价函数。

## 它和降维的关系

原则降维是降维的一种。

它不是把高维原则压扁成一个低级任务，而是让高维原则找到进入现实的接口。

```text
抽象原则
-> 当前问题里的影响范围
-> 可检查的偏差
-> 可复用的沉淀
-> 当前优化目标
-> 奖励 / 损失信号
-> 最小迭代动作
```

好的降维不是让原则变小，而是让原则变得可观察、可检查、可复用、可迭代。

如果升维让我看见更抽象的判断空间，那么原则降维让我把这个判断重新接回当前现实。

## 它在 AI 协作中的价值

原则降维让 AI 不只是参与讨论，也能参与迭代。

它可以避免一种常见情况：

```text
我们讨论出了很好的原则
-> 但没有检查当前问题
-> 没有变成判断依据
-> 没有变成优化目标
-> 下一次又从头开始
```

原则降维可以把一次讨论转化成后续协作的约束。

它让 AI 后续不只是问“要做什么”，还可以参考：

```text
我们之前形成了什么原则？
这个原则适用于当前场景吗？
它影响当前任务的哪些部分？
当前方案和原则是否一致？
本轮优化的奖励信号是什么？
哪些偏差需要避免？
```

这样，AI 协作就不只是一次性问答，而会逐渐变成有记忆、有标准、有方向的迭代。

## 它的风险

原则降维本身也有风险。

第一，过早固化。一个原则可能还只是当前场景中的候选判断，如果太早写成规则，会限制后续思考。

第二，过度外推。一个原则在当前场景有效，不代表它能支配所有场景。

第三，类比误用。如果原则来自另一个场景，表面相似不等于结构相似。原场景成立的条件，当前问题未必满足。

第四，优化目标过窄。为了让原则可操作，可能会把复杂价值压成单一指标，反而损失原本的判断厚度。

第五，把原则变成僵硬 checklist。原则应该提供判断方向，而不是让所有场景都机械服从。

第六，AI 过度执行原则。AI 可能会把原则当成绝对命令，而不是一个需要结合上下文校准的判断标准。

第七，AI 顺着类比说得太合理。当我提出“这个原理是不是也适用于当前问题”时，AI 可能会为了配合这个方向，把迁移关系说得过于顺畅，而没有充分指出失效条件。

## 更好的使用方式

当一个原则出现时，可以先不急着让 AI 直接执行，而是问：

```text
这个原则现在有多稳定？
它来自当前讨论，还是来自别的场景？
如果来自别的场景，当前问题是否满足它的适用条件？
它影响当前问题的哪些部分？
哪些地方明显不一致？
它是否值得沉淀为后续判断依据？
在当前问题里，它应该转化成什么优化目标？
什么算奖励信号，什么算损失信号？
下一步最小可验证动作是什么？
```

也可以用一个更短的循环：

```text
原则出现
-> 判断适用条件
-> 找影响范围
-> 扫描偏差
-> 判断是否沉淀
-> 定义优化目标
-> 做一次最小迭代
-> 根据反馈修正原则
```

这样，原则既不会停留在抽象语言里，也不会被过早固化成僵硬规则。

## 还没想清楚的问题

什么时候一个原则足够稳定，可以被固化？

什么时候原则只是当前场景的临时判断，不应该外推？

当原则来自另一个场景时，如何判断它是结构相似，而不是表面相似？

如何区分“原则降维”和“过度流程化”？

优化目标应该如何保留原则背后的复杂价值，而不是压成单一指标？

AI 应该如何使用原则：作为硬规则、参考约束，还是候选判断？

一个原则进入当前现实后，如何知道它真的产生了约束力？
