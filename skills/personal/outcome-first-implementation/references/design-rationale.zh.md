# 工程复杂度是否仍应是 Agentic Coding 的核心考虑因素？

我的答案是：不应该，至少不应该继续以传统软件开发中的含义和权重被考虑。

Agentic Coding 已经改变了开发中的基本分工。过去由人类工程师承担的大量编码、技术学习、方案尝试和重复修改，现在都可以交给 Agent 完成。实现能力和实现成本已经发生了结构性的变化。

但我发现，Agent 在进行技术选型和 trade-off 时，并没有完全跟上这种开发范式的转换。

## 实现已经进入新范式，trade-off 还没有

有一次，我让 AI 实现一个前端动画效果。它给了我多条技术路线，也像一名有经验的人类工程师一样，比较了代码量、实现难度、技术熟悉度和维护成本。最终，它明显更倾向于一条容易实现、工作量更小，但视觉效果也更保守的路线。

我当时回复：

> 不用考虑技术和实现的复杂度，可以使用任意技术栈，仅以前端视觉效果为核心目标。

我是在明确要求 Agent 不要把自己的实现工作量带入这次视觉方案的选择。

这件事让我清楚地看到一个错位：Agent 的执行方式已经进入了 Agentic Coding，方案选择却仍然沿用过去开发范式中的工程习惯。

它仍然会把需要多写多少代码、技术是否熟悉、实现路径是否足够短作为重要约束；仍然会默认遵循 KISS、渐进式开发和 MVP；仍然会为了降低实现工作量，主动收缩目标或者选择效果上更保守的方案。

## 旧工程习惯在替什么定价

在传统软件开发中，这些习惯是合理的。

代码由人来写，技术需要人来学习，调试会持续消耗人的时间和注意力。一个更复杂的实现意味着更长的排期、更高的沟通成本，以及更多可能无法完成的工作。因此，工程复杂度不仅描述技术本身，也在替有限的人力、时间和认知资源定价。

KISS、渐进式开发和 MVP，很多时候正是在这种资源条件下形成的选择。它们不仅关心系统是否简单，也在帮助团队减少需要亲手完成的工作量。

Agentic Coding 改变了这个前提。当大部分实现由 Agent 承担后，多写几百行代码、多尝试几条路线，或者采用一种人类工程师并不熟悉的技术，不再天然对应过去那种人力成本。

如果 Agent 仍然按照原来的权重处理这些因素，它实际上是在用人类工程师的成本模型，为自己的执行工作定价。然后，它又把这笔未经我要求的成本带入方案选择，替我降低目标。

这就是我看到的成本模型错位：

> 实现能力已经 Agentic 化，方案选择仍以人类工时为中心。

## 范式转换不仅是写代码更快

Agentic Coding 的变化并不只是把同一个工程师变得更快，而是重新划分了人在开发中的位置。

过去，我既要决定想要什么，也要持续计算自己或团队是否能够把它做出来。目标设计和实现预算始终纠缠在同一个决策中。

现在，我可以把更多注意力放在目标、体验、边界和验收标准上，把方案搜索和具体实现交给 Agent。我不再需要因为 Agent 要多写代码、反复尝试或者使用一种陌生技术，就预先缩小自己的目标。

这带来一个明确的决策原则：如果更多 Agent 实现工作能够换来实质上更好的结果，就应优先选择更好的结果。Agent 自己需要承担的执行工作，不应该默认成为产品约束。

## 工程复杂度需要被重新拆开

因此，问题不是简单地保留或删除“工程复杂度”这个考虑因素，而是不能再把它当成一个不可分割的整体。

有一类复杂度不会传播到最终结果：Agent 需要搜索更多方案、生成和重写中间实现、使用人类工程师不熟悉的工具，或者进行更多轮修改。只要这些工作没有改变用户关心的结果，就应由 Agent 在内部吸收。

另一类复杂度会继续进入最终系统：它会影响运行性能、验证难度、用户体验、集成、安全、可逆性或者长期演化。这些不是实现者是否辛苦的问题，而是结果本身发生了变化，当然仍然属于方案决策。

Agentic Coding 重写的是前一类复杂度的价格，并没有取消后一类复杂度的后果。

这也重新定义了 KISS 和 MVP 的位置。简单不应该在比较结果之前就成为默认目标，而应在结果实质相同时成为选择依据。MVP 不应被用来减少 Agent 的实现工作，而应在需要验证需求、审美方向、技术假设或风险时，作为获取信息的工具。

## 为什么形成这个 Skill

我创建 `outcome-first-implementation`，就是为了覆盖这个已经与 Agentic Coding 不再匹配的默认成本模型。

这个 Skill 要求 Agent 在设计和实现时，首先优化我明确声明的结果，而不是默认优化实现者的便利。只由 Agent 承担、不会传播到最终结果中的复杂度，应由它自行吸收；真正会改变结果的复杂度，再进入我需要参与的 trade-off。

对我而言，这不是某一次前端动画任务的特殊要求，而是 Agentic Coding 应有的协作方式：

> 不要继续按照传统人类工时的价格计算 Agent 自己的执行工作，也不要因为实现更麻烦，就未经请求地把更容易完成的结果替换成我真正想要的结果。
