# 一份文档是怎样慢慢变成“墓碑式文档”的

我在长期使用 Agent 维护项目时，反复看到同一种变化。一个项目刚开始时，文档通常很干净。它告诉后来接手项目的人，系统是什么、如何运行、有哪些约束。

然后项目出了第一个问题。Agent 修复了代码，也顺手在文档里补了一句：“不要这样做，因为曾经导致过某个错误。”后来一个方案被放弃，文档里又多了一段解释为什么不再采用旧方案的文字。某次误解被澄清后，又给后续接手的 Agent 留下一条警告。

每一次修改单独看都很合理。它保存了经验，也像是在防止同样的问题再次发生。真正的问题要等时间一长才显现：新内容不断追加，已经失效的旧内容却很少被重写或清理。

系统里的问题已经解决，文档却还留着当时的伤疤。

## 墓碑越来越多以后

当这种维护方式持续下去，文档会渐渐以过去的问题为中心。它记录哪里曾经出错、哪个方案已经死亡、某个 Agent 曾经误解了什么，以及以后千万不要再做什么。

我把这种形态称为“墓碑式文档”。一块墓碑对应一个已经解决的问题或已经废弃的方案。墓碑越来越多，活着的系统反而被挤到角落里。

后续 Agent 读到这些内容时，很难只凭文字判断它们今天是否仍然适用。过去的修复说明可能被当成今天仍然有效的禁令，已经废弃的方案仍占着篇幅和注意力，针对一次性误解写下的警告则可能演变成长期规则。新的 Agent 为了显得稳妥，又会围绕这些旧文字补充更多解释和例外。

最后，Agent 读到的上下文越来越多，却越来越难判断现在到底哪些内容仍然有效。我把这种现象叫作 Contextual Entropy：上下文在增长，当前信号却越来越难辨认。

## 把当前与历史分开

项目仍然需要保留历史。架构决策、事故复盘、迁移过程、纠偏记录和版本变化都可能有长期价值。

真正需要分开的是两件事：一段历史记录是否值得保存，以及它是否应该默认进入 Agent 的工作上下文。

> 要不要保存，取决于存档价值；要不要默认放进上下文，取决于协作需要。

最好的清理并不是定期搬运这些墓碑，而是在内容写入时就判断：它表达的是一条当前规则，还是只是在记录一次已经完成的修复？

当前文档应该始终说明系统现在的状态：系统如何运行，哪些决定仍然成立，哪些约束还在，哪些问题尚未解决。系统发生变化时，文档也要随之重写，而不是在旧叙述后面再加一层补丁。

值得保留的历史，可以单独写进 ADR、postmortem、changelog、migration 记录或 `docs/history/` 等路径。需要时再查，不必每次做新设计或写新代码时都默认加载。如果某段历史仍然影响当前决策，当前文档保留一条简短规则或链接就够了。

也不是每一次纠偏都值得归档。记录一次性误解的文字、已经失效的警告和没有后续价值的旧内容，都可以直接删除；仍然有效的经验，应提炼成当前规则；只有对追溯、复盘、迁移或决策仍有价值的记录，才值得归档。

把所有错误集中写进一个 `historical-errors.md`，只会在另一个位置重新建造墓园。历史应该按用途保存，而不是因为不敢删除就全部收集起来。

我后来更愿意把这项偏好称为 `living-documentation`，而不是强调所谓的“当前真相”。我关心的并不是让文档宣称自己永远正确，而是让它能够随着系统一起变化：该重写时重写，该删除时删除，该归档时归档。

这样的做法并不会让项目失去记忆。当前文档保持清楚，历史记录保留来路；只有确有需要时，才把两者联系起来，而不是任由惯性将它们混在一起。

我希望当前文档能让后来接手项目的人迅速了解系统的状态和运行方式，也能让 Agent 在后续任务中获得同样清楚的起点。需要调查事故或理解旧决策时，相应的历史记录也应该很容易找到。每次开始新任务，都不该先穿过一片由过去错误组成的墓园。
