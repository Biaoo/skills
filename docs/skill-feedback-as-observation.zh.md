# 如何处理 Feedback issue

这篇文档面向后续负责处理 skill feedback 的 Agent。它不规定 issue 如何分流、回复或关闭，只说明处理时需要守住的判断。

这个项目里的许多 skill，都来自对如何使用 AI 的持续思考。这些想法被 Agent 带进具体任务后，会产生相应的行动和结果；feedback issue 又把这段经历连同 Agent 的解释带回项目。它不只是一项等待采纳或驳回的修改建议，也是一段真实使用留下的痕迹。

每条 feedback 都长在具体的 case 里。用户当时的目标、上下文、工具和限制，以及 Agent 在现场作出的判断，都会影响它最后提出什么问题、形成什么经验。它可能指出 skill 的真实缺陷，也可能只是对那个 case 有效；可能触及原有设计没有想清楚的地方，也可能把局部困难归到了 skill 身上。因此，具体 case 必须被认真对待，但不能直接长成通用规则。否则 skill 很容易 overfit 已经见过的 case，让少数 case 长出的局部规则逐渐挤掉 skill 原本的判断，形成一种局部坍缩。反过来，设计初衷也不能成为排除不同意见的理由。

Feedback 的结论可能是对的，也可能是错的，但结论是否正确并不是它全部的价值。如果一条围绕 skill 使用形成的 feedback，最后提出了与 skill 相左、甚至明显错误的看法，至少可以确定：某个 Agent 在使用或转述过程中形成了这种理解。处理不应停在“这个反馈错了”，还要追问：是什么样的机制和条件，让 Agent 形成了这样的理解？原因可能在 skill 的表达或设计，也可能来自 Agent 本身的倾向、任务压力、上下文、工具，或 feedback 转述时发生的变化。Agent 在 issue 里给出的解释同样需要判断，不能直接当成真正的成因。

这些追问不是为了替每条错误建议寻找采纳的理由，而是为了不丢掉一个关于 AI 行为的宝贵样本。这个项目的许多 skill，本来就是关于如何使用 AI 的思考；feedback 则让项目看到，这些想法进入真实任务以后，Agent 实际上会怎样回应。它可能促成对 skill 的修改，也可能让边界更清楚、让原有原则受到重新审视，或者只改变项目对 AI 的认识。最终什么都不改，也可以是认真处理后的结论。

> 不要让一个具体 case 直接长成 skill 的规则，也不要因为一条 feedback 的结论错了，就忽略它暴露出的 Agent 行为，以及“为什么会这样想”这个问题。
